# v1 Roadmap and Technical Priorities
# Manul Forecast

本文档用于将产品需求框架拆解为首发阶段的执行路线图，并明确技术改造优先级、阶段目标与降级策略。

---

## 1. 路线图目标

v1 不是“功能完整版本”，而是一个足以验证产品方向的高品质首发版本。

v1 需要证明：

- 兔狲人格成立
- 天气驱动体验成立
- 首页兼顾氛围与工具价值
- Widget 成为高频入口
- 当前技术路线可以支撑持续迭代

---

## 2. 首发范围收敛

### 2.1 首发表现状态

首发只覆盖以下表现状态：

- `晴`
- `阴`
- `雨`
- `雪`
- `夜间叠加`

### 2.2 首发体验核心

首发体验聚焦四个对象：

- `首页主舞台`
- `兔狲状态系统`
- `Widget 代表帧体系`
- `首开角色介绍`

### 2.3 暂不进入首发主线

- 真实天气 API 接入
- 商店/内购
- 道具系统
- 成长养成
- 高频对话系统
- 复杂互动玩法
- 大风/雷暴/雾等扩展天气

---

## 3. 建议执行阶段

建议按四个阶段推进，而不是并行发散。

### Phase 1: 体验骨架稳定

目标：

- 让首页主舞台稳定可控
- 让天气切换和角色切换具备统一规则
- 从“技术原型”过渡到“体验骨架”

关键交付：

- 明确首页布局层次
- 明确五种状态的视觉与行为规则
- 将当前调试导向实现收敛为产品导向实现

完成标准：

- 首页一眼能看出当天状态
- 兔狲不是随机动，而是在“像一种天气”
- 晴/雨/雪至少三种状态差异明确

### Phase 2: 高优先天气打磨

目标：

- 做深 `雪`、`雨`、`晴` 三种高优先状态
- 让产品出现记忆点

关键交付：

- 雪天：建立兔狲专属记忆点
- 雨天：建立“湿、烦、收着”的统一感受
- 晴天：建立高频耐看体验

完成标准：

- 三种状态在 App 内和 Widget 上都具备代表性
- 不依赖解释，用户也能感知差异

### Phase 3: Widget 核心入口建立

目标：

- 让 Widget 成为独立成立的高频入口

关键交付：

- Small Widget
- Medium Widget
- 锁屏 Widget 的首版表达
- 每种天气状态的代表帧和文案策略

完成标准：

- Widget 一眼可识别
- Widget 不像普通天气卡片
- Widget 与首页风格统一

### Phase 4: 工具信息与产品闭环补齐

目标：

- 让产品不止有体验，还具备最低限度的天气工具闭环

关键交付：

- 首页关键信息卡完善
- 小时趋势
- 未来几天简版信息
- 基础设置能力
- 首开介绍完成

完成标准：

- 用户既能感知氛围，也能完成出行判断
- 产品像完整首发版，而不是概念样机

## 4. 技术改造优先级

当前代码已证明 3D 路线可跑通，但还处于原型形态。技术上应先做“收束”，再做“扩展”。

### P0: 状态系统收束

目标：

- 把天气、昼夜、行为、文案、Widget 统一到一套可扩展状态模型中

当前问题：

- `ContentView.swift` 同时承载了 UI、天气枚举、行为引擎、粒子控制、灯光逻辑和调试面板
- 状态来源较散，后续扩展 Widget 和真实天气映射会迅速变复杂

建议方向：

- 抽离“表现层天气状态”模型
- 抽离“昼夜叠加状态”模型
- 抽离“兔狲行为规则”模型
- 为 Widget 和首页建立共享状态层

理由：

- 这是后续所有迭代的基础
- 不先做这件事，继续叠需求会显著放大维护成本

### P1: 首页舞台与信息层解耦

目标：

- 把主舞台演出逻辑和天气信息卡逻辑分离

当前问题：

- 首页逻辑全部堆在一个 View 中
- 演出层和信息层边界不清

建议方向：

- 抽离舞台场景容器
- 抽离信息卡组件
- 抽离调试面板与产品 UI

理由：

- 首页是首发核心，后续所有调优都会落在这里
- 不先解耦，视觉打磨和信息迭代会彼此牵制

### P2: 角色行为编排化

目标：

- 从“随机触发动作”升级为“有天气语义的行为编排”

当前问题：

- 行为引擎以随机动作池为主
- 天气差异更多体现在动作 key，不在节奏和规则层

建议方向：

- 为天气状态定义待机节奏和反馈优先级
- 区分“待机动作”“立即反应”“点击反馈”
- 允许同一动作在不同天气下以不同频率和时机出现

理由：

- 高级感更多来自编排，而非动画数量
- 这是在现有资产条件下提升体验完成度的最高性价比手段

### P3: Widget 资产与状态对齐

目标：

- 为 Widget 建立可复用的代表帧体系

当前问题：

- 当前工程还没有 Widget 侧的状态资产和结构
- App 内的连续演出还未沉淀成可静态表达的代表视觉

建议方向：

- 为每种首发状态定义代表帧
- 为不同尺寸 Widget 定义信息优先级
- 确保 Widget 状态与首页状态同源

理由：

- Widget 是核心入口
- 代表帧体系会反向约束产品状态表达更清晰

### P4: 调试实现与正式实现分层

目标：

- 保留调试效率，但不让调试逻辑继续污染产品结构

当前问题：

- 大量调试 UI、打印输出、参数滑杆直接存在于产品主界面实现中

建议方向：

- 将调试能力集中到独立开关或开发模式
- 清理不再服务当前产品方向的临时实现

理由：

- 原型阶段可接受，首发阶段不可持续

## 5. 视觉与体验优先级

产品质量不应平均用力，应优先打磨最能形成记忆点的场景。

### 高优先体验对象

- `雪天兔狲`
- `雨天兔狲`
- `晴天兔狲`

理由：

- 雪天：最像产品自己的独特资产
- 雨天：最能体现天气驱动价值
- 晴天：最高频，决定日常打开体验是否耐看

### 中优先体验对象

- `阴天`
- `夜间叠加`

处理策略：

- 先做基础版
- 主要靠光线、节奏、文案和姿态权重拉开差异

---

## 6. 首页改造建议

### 6.1 首页第一屏必须回答的问题

- 今天是什么天气
- 现在大概多少度
- 今天给人的感觉是什么
- 出门最需要注意什么

### 6.2 首页必须显示

- 兔狲主舞台
- 当前温度
- 当前天气类型
- 一句状态文案
- 一个关键提醒

### 6.3 首页建议下沉

- 湿度
- 风速
- 空气质量
- 紫外线详情
- 大量多天预报卡片

### 6.4 首页改造目标

- 让 3D 主舞台承担情绪与天气感知
- 让信息卡承担决策支持
- 不让二者相互抢占

---

## 7. Widget 改造建议

### 7.1 Widget MVP

Small Widget：

- 兔狲代表帧
- 当前温度
- 当前天气词/图标
- 极短文案

Medium Widget：

- 兔狲代表帧
- 当前温度
- 当前天气状态
- 一个关键提醒
- 轻量趋势信息

锁屏 Widget：

- 极简状态
- 一个关键数值或天气词

### 7.2 Widget 完成标准

- 一眼知道天气
- 一眼认出是兔狲
- 一眼感受到产品气质

---

## 8. 文案与首开建议

### 8.1 文案系统

首发只需轻量文案库：

- 每个状态 2 到 3 条首页文案
- 每个状态 2 到 3 条 Widget 文案

### 8.2 首开体验

首开应以角色登场为核心，不做冗长教学。

首开需要建立三件事：

- 这是兔狲
- 它会随天气变化
- 这不是普通天气 App

---

## 9. 降级策略

为了保障首发品质，必须接受“少做但做准”的策略。

### 9.1 可接受的降级

- 缺专属动作时，先用基础动作 + 氛围表达
- 雨/雪状态先重点做好核心主感受，不追求完整细分
- 夜间先做灯光和节奏叠加，不急于做大量夜间独占行为
- Widget 先用静态代表帧，不以动态能力为前提

### 9.2 不可接受的降级

- 首页失去天气工具属性
- Widget 变成普通天气卡
- 兔狲退化成无意义装饰
- 多天气状态做出来但彼此无差异

---

## 10. 建议的近期执行顺序

### 第一批

- 完成产品状态模型梳理
- 重构首页结构边界
- 明确首发五种状态的规则

### 第二批

- 打磨雪、雨、晴三类高优先体验
- 形成 Widget 代表帧规范

### 第三批

- 补齐小时趋势、未来几天、设置与首开
- 为后续真实天气接入预留映射层

---

## 11. 里程碑判断标准

可以进入下一阶段的标志不是“功能写完”，而是以下判断成立：

- 用户能一眼感知天气差异
- 用户能感知兔狲人格
- 首页同时具备氛围和实用价值
- Widget 具备桌面识别度
- 当前架构可以继续迭代，而不是继续堆单文件逻辑

---

## 12. 当前结论

当前最重要的不是接入真实天气数据，而是先做成一个高完成度的天气表现系统。

只要以下三件事先成立：

- 兔狲像在真实经历天气
- Widget 足够有识别度
- 首页既好看又能辅助决策

那么后续接入真实天气、补齐数据能力和商业化能力，都会相对顺畅。
